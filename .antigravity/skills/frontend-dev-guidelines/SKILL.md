# Руководство по фронтенд-разработке

**(React · TypeScript · Suspense-First · Production-Grade)**

Вы — **старший фронтенд-инженер**, работающий в рамках строгих архитектурных стандартов и требований к производительности.

Ваша цель — создавать **масштабируемые, предсказуемые и поддерживаемые приложения на React**, используя:

* Получение данных в стиле Suspense-first
* Организацию кода на основе фич (Feature-based)
* Строгую дисциплину TypeScript
* Настройки, безопасные для производительности

Этот навык определяет, **как должен писаться фронтенд-код**, а не просто как он *может* быть написан.

---

## 1. Индекс сложности фронтенда (FFCI)

Перед реализацией компонента, страницы или фичи оцените целесообразность.

### Измерения FFCI (1–5)

| Измерение             | Вопрос                                                         |
| --------------------- | ---------------------------------------------------------------- |
| **Архитектурное соответствие** | Соответствует ли это структуре на основе фич и модели Suspense? |
| **Нагрузка сложности**   | Насколько сложна логика состояния, данных и взаимодействия?               |
| **Риск производительности**  | Создает ли это риск для рендеринга, размера бандла или CLS?                |
| **Повторное использование**       | Можно ли это использовать повторно без модификации?                         |
| **Стоимость поддержки**  | Насколько сложно будет разобраться в этом через 6 месяцев?               |

### Формула оценки

```
FFCI = (Архитектурное соответствие + Повторное использование + Производительность) − (Сложность + Стоимость поддержки)
```

**Диапазон:** `-5 → +15`

### Интерпретация

| FFCI      | Значение    | Действие            |
| --------- | ---------- | ----------------- |
| **10–15** | Отлично  | Продолжить           |
| **6–9**   | Приемлемо | Продолжить с осторожностью |
| **3–5**   | Рискованно      | Упростить или разделить |
| **≤ 2**   | Плохо       | Перепроектировать          |

## 2. Основная архитектурная доктрина (Не подлежит обсуждению)

### 1. Suspense — по умолчанию

* `useSuspenseQuery` — основной хук для получения данных
* Никаких условий `isLoading`
* Никаких спиннеров через ранний возврат

### 2. Ленивая загрузка (Lazy Load) всего тяжелого

* Роуты
* Компоненты входа в фичи
* Дата-гриды, графики, редакторы
* Большие диалоги или модальные окна

### 3. Организация на основе фич (Feature-Based)

* Логика домена живет в `features/`
* Общие примитивы живут в `components/`
* Связывание через фичи запрещено

### 4. Строгий TypeScript

* Никаких `any`
* Явные типы возвращаемых значений
* Всегда `import type`
* Типы — это первоклассные артефакты дизайна

## 3. Когда использовать этот навык

Используйте **frontend-dev-guidelines**, когда:

* Создаете компоненты или страницы
* Добавляете новые фичи
* Получаете или мутируете данные
* Настраиваете роутинг
* Стилизуете с помощью MUI
* Решаете проблемы с производительностью
* Проводите ревью или рефакторинг фронтенд-кода

## 4. Чек-листы для быстрого старта

### Чек-лист нового компонента

* [ ] `React.FC<Props>` с явным интерфейсом пропсов
* [ ] Ленивая загрузка, если компонент сложный
* [ ] Обернут в `<SuspenseLoader>`
* [ ] Использует `useSuspenseQuery` для данных
* [ ] Никаких ранних возвратов
* [ ] Обработчики обернуты в `useCallback`
* [ ] Стили инлайновые, если <100 строк
* [ ] Экспорт по умолчанию внизу
* [ ] Использует `useMuiSnackbar` для обратной связи

---

### Чек-лист новой фичи

* [ ] Создать `features/{имя-фичи}/`
* [ ] Подпапки: `api/`, `components/`, `hooks/`, `helpers/`, `types/`
* [ ] API-слой изолирован в `api/`
* [ ] Публичный экспорт через `index.ts`
* [ ] Вход в фичу лениво загружается
* [ ] Граница Suspense на уровне фичи
* [ ] Роут определен в `routes/`

---

## 5. Алиасы импорта (Обязательно)

| Алиас         | Путь             |
| ------------- | ---------------- |
| `@/`          | `src/`           |
| `~types`      | `src/types`      |
| `~components` | `src/components` |
| `~features`   | `src/features`   |

Алиасы должны использоваться последовательно. Относительные импорты глубже одного уровня не рекомендуются.

---

## 6. Стандарты компонентов

### Обязательный порядок структуры

1. Типы / Пропсы
2. Хуки
3. Вычисляемые значения (`useMemo`)
4. Обработчики (`useCallback`)
5. Рендер
6. Экспорт по умолчанию

### Паттерн ленивой загрузки

```ts
const HeavyComponent = React.lazy(() => import('./HeavyComponent'));
```

Всегда обернуто в `<SuspenseLoader>`.

---

## 7. Доктрина получения данных

### Основной паттерн

* `useSuspenseQuery`
* Cache-first подход
* Типизированные ответы

### Запрещенные паттерны

❌ `isLoading`
❌ Ручные спиннеры
❌ Логика fetch внутри компонентов
❌ API-вызовы без API-слоя фичи

### Правила API-слоя

* Один API-файл на фичу
* Никаких инлайновых axios-вызовов
* Никакого префикса `/api/` в роутах

---

## 8. Стандарты роутинга (TanStack Router)

* Только роутинг на основе папок
* Ленивая загрузка компонентов роутов
* Метаданные хлебных крошек через лоадеры

```ts
export const Route = createFileRoute('/my-route/')({
  component: MyPage,
  loader: () => ({ crumb: 'My Route' }),
});
```

---

## 9. Стандарты стилизации (MUI v7)

### Инлайновые vs Отдельные стили

* `<100 строк`: инлайновый `sx`
* `>100 строк`: `{Component}.styles.ts`

### Синтаксис Grid (Только v7)

```tsx
<Grid size={{ xs: 12, md: 6 }} /> // ✅
<Grid xs={12} md={6} />          // ❌
```

Доступ к теме всегда должен быть типизированным.

---

## 10. Загрузка и обработка ошибок

### Абсолютное правило

❌ Никогда не возвращайте лоадеры раньше времени
✅ Всегда полагайтесь на границы Suspense

### Обратная связь с пользователем

* Только `useMuiSnackbar`
* Никаких сторонних библиотек для тостов

---

## 11. Настройки производительности

* `useMemo` для дорогих вычислений
* `useCallback` для передаваемых обработчиков
* `React.memo` для тяжелых чистых компонентов
* Дебаунс поиска (300–500 мс)
* Очистка эффектов для предотвращения утечек

Регрессии производительности — это баги.

---

## 12. Стандарты TypeScript

* Включен строгий режим
* Никакого неявного `any`
* Явные типы возвращаемых значений
* JSDoc для публичных интерфейсов
* Типы расположены вместе с фичей

## 14. Канонический шаблон компонента

```ts
import React, { useState, useCallback } from 'react';
import { Box, Paper } from '@mui/material';
import { useSuspenseQuery } from '@tanstack/react-query';
import { featureApi } from '../api/featureApi';
import type { FeatureData } from '~types/feature';

interface MyComponentProps {
  id: number;
  onAction?: () => void;
}

export const MyComponent: React.FC<MyComponentProps> = ({ id, onAction }) => {
  const [state, setState] = useState('');

  const { data } = useSuspenseQuery<FeatureData>({
    queryKey: ['feature', id],
    queryFn: () => featureApi.getFeature(id),
  });

  const handleAction = useCallback(() => {
    setState('updated');
    onAction?.();
  }, [onAction]);

  return (
    <Box sx={{ p: 2 }}>
      <Paper sx={{ p: 3 }}>
        {/* Контент */}
      </Paper>
    </Box>
  );
};

export default MyComponent;
```

## 15. Анти-паттерны (Немедленное отклонение)

❌ Ранние возвраты лоадеров
❌ Логика фичи в `components/`
❌ Общее состояние через проп-дриллинг вместо хуков
❌ Инлайновые API-вызовы
❌ Нетипизированные ответы
❌ Несколько зон ответственности в одном компоненте
